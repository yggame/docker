安装 NVIDIA Container Toolkit

官方安装文档：NVIDIA Container Toolkit Install Guide

以下流程适用于 Ubuntu、Debian 以及常见 Debian 衍生发行版。

安装基础依赖

bash
sudo apt-get update && sudo apt-get install -y --no-install-recommends \
   ca-certificates \
   curl \
   gnupg2

添加 NVIDIA 软件源

bash
distribution=$(. /etc/os-release;echo $ID$VERSION_ID) \
   && curl -s -L https://nvidia.github.io/nvidia-docker/gpgkey | sudo apt-key add - \
   && curl -s -L https://nvidia.github.io/nvidia-docker/$distribution/nvidia-docker.list | sudo tee /etc/apt/sources.list.d/nvidia-docker.list

配置稳定版本仓库

bash
curl -fsSL https://nvidia.github.io/libnvidia-container/gpgkey | sudo gpg --dearmor -o /usr/share/keyrings/nvidia-container-toolkit-keyring.gpg \
  && curl -s -L https://nvidia.github.io/libnvidia-container/stable/deb/nvidia-container-toolkit.list | \
    sed 's#deb https://#deb [signed-by=/usr/share/keyrings/nvidia-container-toolkit-keyring.gpg] https://#g' | \
    sudo tee /etc/apt/sources.list.d/nvidia-container-toolkit.list

可选：启用 experimental 包

bash
sudo sed -i -e '/experimental/ s/^#//g' /etc/apt/sources.list.d/nvidia-container-toolkit.list

更新并安装指定版本

bash
sudo apt-get update
bash
export NVIDIA_CONTAINER_TOOLKIT_VERSION=1.19.1-1
sudo apt-get install -y \
  nvidia-container-toolkit=${NVIDIA_CONTAINER_TOOLKIT_VERSION} \
  nvidia-container-toolkit-base=${NVIDIA_CONTAINER_TOOLKIT_VERSION} \
  libnvidia-container-tools=${NVIDIA_CONTAINER_TOOLKIT_VERSION} \
  libnvidia-container1=${NVIDIA_CONTAINER_TOOLKIT_VERSION}

配置并重启 Docker 服务

让 Docker 重新加载 NVIDIA 运行时和 CDI 配置。

自动配置 Docker 运行时
bash
sudo nvidia-ctk runtime configure --runtime=docker
生成新版 Docker 识别 Blackwell 显卡所需的 CDI 配置文件
bash
sudo nvidia-ctk cdi generate --output=/etc/cdi/nvidia.yaml
重启 Docker 服务让配置生效
bash
sudo systemctl restart docker

常用 Docker 运行命令

所有桥梁都搭建完毕后，可以按需选择下面的运行模板。

vLLM nightly 基础进入命令

bash
docker run -it --entrypoint /bin/bash --gpus all \
  -u $(id -u):$(id -g) \
  -v /hdd:/hdd \
  -v /hdd2:/hdd2 \
  -v /home/your_user/.cache/huggingface:/root/.cache/huggingface \
  vllm/vllm-openai:nightly-x86_64

带用户识别与 Hugging Face 缓存映射

bash
docker run -it \
  --entrypoint /bin/bash \
  --gpus all \
  -e NVIDIA_DISABLE_REQUIRE=true \
  -u $(id -u):$(id -g) \
  -v /etc/passwd:/etc/passwd:ro \
  -v /etc/group:/etc/group:ro \
  -v /hdd:/hdd \
  -v /hdd2:/hdd2 \
  -v /hdd/.cache/huggingface:/root/.cache/huggingface \
  --env HF_HOME=/root/.cache/huggingface \
  baichuan-env:v1

把 HOME 与 HF_HOME 显式放到 /hdd

bash
docker run -it \
  --entrypoint /bin/bash \
  --gpus all \
  -e NVIDIA_DISABLE_REQUIRE=true \
  -u $(id -u):$(id -g) \
  -v /etc/passwd:/etc/passwd:ro \
  -v /etc/group:/etc/group:ro \
  -v /hdd:/hdd \
  -v /hdd2:/hdd2 \
  -e HOME=/hdd \
  --env HF_HOME=/hdd/huggingface \
  baichuan-env:v1

带共享内存、内存限制与自定义提示符

bash
docker run -it \
  --entrypoint /bin/bash \
  --gpus all \
  -u $(id -u):$(id -g) \
  -v /etc/passwd:/etc/passwd:ro \
  -v /etc/group:/etc/group:ro \
  -v /hdd:/hdd \
  -v /hdd2:/hdd2 \
  -e HOME=/hdd/DockerHome/your_user \
  --env HF_HOME=/hdd/huggingface \
  --shm-size=64g \
  --memory=128g \
  baichuan-env:v1 \
  -c '/bin/bash --rcfile <(echo "export PS1=\"\[\033[01;32m\][🐉 DOCKER-ENV]\[\033[00m\] \u@\h:\w\\$ \"")'

关键参数说明

-u $(id -u):$(id -g)让容器内进程以当前普通用户身份运行。
-v /etc/passwd:/etc/passwd:ro让容器内部正确识别用户名，避免显示 I have no name!。
--env HF_HOME=...指定模型权重下载目录，避免写入虚拟 /root 时出现权限错误。
-e NVIDIA_DISABLE_REQUIRE=true让 Docker 忽略部分驱动版本要求并尝试强制启动。

Docker 解析逻辑

  • docker run [OPTIONS]：前面的 -v、-e、--shm-size 都是配置项。
  • IMAGE：例如 baichuan-env:v1 是分界线，表示配置结束，接下来加载这个镜像。
  • [COMMAND] [ARG...]：镜像名后面的 -c '/bin/bash ...' 会被识别为命令。因为前面指定了 --entrypoint /bin/bash，最终执行的是 /bin/bash -c '/bin/bash --rcfile ...'。

保存镜像

常见方式有三种：快速保存当前容器、用 Dockerfile 规范构建、用 build 命令生成新镜像。

方法一：使用 docker commit 保存镜像

查看容器 ID

在宿主机终端运行以下命令，找到正在修改的容器 CONTAINER ID。

bash
docker ps
docker ps -a
执行 commit 保存
bash
docker commit <容器ID> <新镜像名称>:<版本标签>
bash 示例
docker commit a1b2c3d4e5f6 my-pytorch:v1
验证与使用
bash
docker images
docker run -it my-pytorch:v1 /bin/bash

方法二：Dockerfile 构建（推荐）

docker commit 很方便，但它是一个“黑盒”。更规范的做法是把修改步骤写入 Dockerfile，让 Docker 自动构建。

Dockerfile
# 1. 指定基础镜像
FROM pytorch/pytorch:2.8.0-cuda12.6-cudnn9-runtime

# 2. 设置工作目录
WORKDIR /workspace

# 3. 写入环境安装命令
RUN pip install --no-cache-dir pandas scikit-learn

# 4. 安装系统工具
RUN apt-get update && apt-get install -y tmux

方法三：运行 build 命令生成新镜像

在 Dockerfile 所在目录下执行。末尾的点 . 表示当前目录。

bash
docker build -t my-pytorch:v2 .

删除容器

先查看状态，运行中的容器需要先停止，再删除。

bash
docker ps -a
docker stop <container_id>
docker rm <container_id>

如果需要批量删除所有已停止的容器：

bash
docker container prune

删除镜像

bash
docker images
docker rmi <image_id>

如果镜像被容器占用，需要先停止并删除相关容器。强制删除不推荐，但命令如下：

bash
docker rmi -f <image_id>
删除前确认没有重要容器依赖该镜像；使用 -f 时尤其谨慎。

安全加固

这里整理两道防线：从 Docker 守护进程层面限制提权，再通过宿主机 wrapper 脚本拦截高危参数。

第一道防线：Docker 守护进程配置

直接修改宿主机 Docker 配置文件 /etc/docker/daemon.json，从底层锁住高危行为。

json
{
  "no-new-privileges": true,
  "userns-remap": "default",
  "live-restore": true
}
bash
sudo systemctl restart docker
no-new-privileges禁止容器内进程获得比当前更多的特权，降低 SUID 或漏洞提权风险。
userns-remap把容器内 root 映射为宿主机普通用户，降低乱挂载敏感目录后的破坏能力。
live-restoreDocker 守护进程重启时，后台运行容器仍可保持运行。

第二道防线：宿主机拦截脚本

编写 Docker wrapper，放到 /usr/local/bin/docker，利用环境变量优先级在用户输入命令时先做安全审查。

bash
sudo vim /usr/local/bin/docker
bash wrapper
#!/bin/bash

# 定义禁止使用的危险参数黑名单
BLACKLIST=("--privileged" "--net=host" "--net host" "--pid=host" "--pid host")

# 检查用户输入的参数中是否包含黑名单内容
for arg in "$@"; do
  for banned in "${BLACKLIST[@]}"; do
    if [[ "$arg" == *"$banned"* ]]; then
      echo "安全错误: 检测到高危参数 [$banned]，已被系统拒绝启动！"
      echo "提示：普通用户禁止使用特权模式、主机网络及主机进程共享。"
      exit 1
    fi
  done
done

# 如果通过检查，调用真正的 Docker 执行
exec /usr/bin/docker "$@"
bash
sudo chmod +x /usr/local/bin/docker
效果：当 Docker 组用户尝试运行 docker run --privileged ... 时，脚本会第一时间切断请求，避免传给 Docker 后台。

容器保存并重新加载

如果是老服务器中途开启安全模式，通常可以按下面流程迁移镜像。

临时关闭安全模式。
将需要的旧镜像导出为文件。
bash
docker save -o my_image.tar pytorch/pytorch:xxxx
开启安全模式，重新加上参数并重启 Docker。
将镜像导入到安全模式的新目录中。
bash
docker load -i my_image.tar
